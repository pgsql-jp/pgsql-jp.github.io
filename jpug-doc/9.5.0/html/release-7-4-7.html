E.223. リリース7.4.7

<title>Release 7.4.7</title>

リリース日

<title>Release Date</title>

2005-01-31

This release contains a variety of fixes from 7.4.6, including several security-related issues. For information about new features in the 7.4 major release, see <xref linkend="release-7-4">.
このリリースは複数のセキュリティ問題を含む、7.4.6の各種不具合を修正したものです。 7.4メジャーリリースにおける新機能については「リリース 7.4」を参照してください。

E.223.1. バージョン7.4.7への移行

<title>Migration to Version 7.4.7</title>

A dump/restore is not required for those running 7.4.X.
7.4.Xからの移行の場合は ダンプ/リストアは必要ありません。

E.223.2. 変更点

<title>Changes</title>
<listitem><para>Disallow <command>LOAD</> to non-superusers</para>

  • スーパーユーザ以外のLOADを不許可としました

    On platforms that will automatically execute initialization functions of a shared library (this includes at least Windows and ELF-based Unixen), <command>LOAD</> can be used to make the server execute arbitrary code. Thanks to NGS Software for reporting this.</para></listitem>
    共有ライブラリの初期化関数を自動的に行うプラットフォーム(少なくともWindowsやELFベースのUnixを含みます)では、LOADを使用して任意のコードを実行させることができました。 この問題を報告したNGSソフトウェアに感謝します。

    • <listitem><para>Check that creator of an aggregate function has the right to execute the specified transition functions</para>

  • 集約関数の作成者が指定した遷移関数を実行する権利を持つかどうかを検査します。

    This oversight made it possible to bypass denial of EXECUTE permission on a function.</para></listitem>
    この見落としのため、関数のEXECUTE権限を回避することができてしまいました。

    • <listitem><para>Fix security and 64-bit issues in contrib/intagg</para></listitem>

  • contrib/intaggにおけるセキュリティと64ビット問題を修正しました。

    • <listitem><para>Add needed STRICT marking to some contrib functions (Kris Jurka)</para></listitem>

  • 一部のcontrib関数で必要であったSTRICT印を追加しました。(Kris Jurka)

    • <listitem><para>Avoid buffer overrun when plpgsql cursor declaration has too many parameters (Neil)</para></listitem>

  • plpgsqlのカーソル定義のパラメータが多すぎた場合のバッファオーバーランを防止しました。 (Neil)

    • <listitem><para>Fix planning error for FULL and RIGHT outer joins</para>

  • FULLおよびRIGHT外部結合用の計画作成エラーを修正しました。

    The result of the join was mistakenly supposed to be sorted the same as the left input. This could not only deliver mis-sorted output to the user, but in case of nested merge joins could give outright wrong answers.
    結合結果が間違って、左側入力と同じようにソートされたものとみなされていました。 これはユーザに間違ったソート結果を導くだけでなく、入れ子状マージ結合の場合は全く間違った回答を与えるかもしれませんでした。

  • Fix plperl for quote marks in tuple fields

  • Fix display of negative intervals in SQL and GERMAN datestyles

  • Make age(timestamptz) do calculation in local timezone not GMT
